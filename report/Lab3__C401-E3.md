## A. Group Report

### 1. Executive Summary

Project goal: build a ReAct travel assistant for multi-step planning (weather + flights + budget), then compare it against a chatbot baseline.

- **Success Rate**
  - Agent: 55/56 sessions reached valid `final_answer` (~98.2%).
  - Chatbot: 4/4 sessions returned an answer (100%).
- **Key Outcome**
  - Agent performs better on multi-step scenarios thanks to tool usage, structured traces, and citations.

### 2. System Architecture & Tooling

#### 2.1 ReAct Loop

1. `Thought` - model reasons about next step.
2. `Action` - model calls exactly one tool.
3. `Observation` - tool returns structured JSON.
4. Repeat until `Final Answer` or `max_steps`.

#### 2.2 Tool Inventory

| Tool | Input | Use case |
| :--- | :--- | :--- |
| `get_weather` | `city` | Current weather + short forecast with resilient city query handling. |
| `search_flights` | `origin, destination, departure_date` | One-way flight search (crawl-first with fallback). |
| `search_roundtrip_flights` | `origin, destination, departure_date, return_date` | Round-trip search from two one-way searches. |
| `search_itinerary_flights` | `segments_text` | Multi-leg itinerary search and aggregation. |
| `calculate_travel_budget` | budget/cost fields | Budget feasibility computation. |

#### 2.3 LLM Providers

- **Primary**: Gemini (`gemini-2.5-flash`)
- **Secondary**: provider factory supports OpenAI/local fallback paths.

### 3. Telemetry & Performance Dashboard

Source: `report/exports/*.csv` generated from `logs/`.

- **Total sessions**: 60 (56 agent, 4 chatbot)
- **Agent P50 latency**: ~15,316.5 ms/session
- **Observed max latency**: ~82,272 ms
- **Average prompt tokens**
  - Agent: ~11,751.2

### 4. RCA - Failure Traces

#### 4.1 Success trace (multi-step)

- **Input**: HAN -> DAD + weather + budget query
- **Trace**
  - Action 1: `get_weather(Da Nang, VN)`
  - Action 2: `search_flights(HAN, DAD, 2026-04-15)`
  - Action 3: `calculate_travel_budget(...)`
  - Final: budget feasibility conclusion using tool outputs

#### 4.2 Failure trace (parser mismatch)

- **Input**: `get_weather(Da Nang, VN)` and `search_flights(origin='SGN', ...)`
- **Observation**
  - weather used to fail with `expected 1 args, got 2`
  - flight used to fail due to single-quote tokens
- **Root cause**
  - comma splitting for one-arg tool
  - inconsistent quote stripping
- **Fix**
  - one-param tool keeps full string
  - normalize both single and double quoted arguments
  - regression tests added in `tests/test_travel_tools.py`

### 5. Ablation Studies & Experiments

#### 5.1 Prompt v1 vs v2

- **Diff**: stricter tool-call guardrails, anti-hallucination instructions, better roundtrip/itinerary routing.
- **Result**: fewer invalid tool calls, improved stability on weather-only and multi-step prompts.

#### 5.2 Chatbot vs Agent

| Case | Chatbot | Agent | Winner |
| :--- | :--- | :--- | :--- |
| Simple weather | Fast, generic | Tool-backed with citation | Agent |
| Multi-step planning | Tends to miss evidence | Structured weather -> flight -> budget chain | Agent |
| Complex itinerary | Limited decomposition | Multi-leg tool path with aggregation | Agent |

### 6. Flowchart & Group Insights

```text
User query
   |
   v
Thought + Action selection
   |
   v
Run one tool (weather / flight / budget)
   |
   v
Observation (JSON)
   |
   +--> enough data? -- yes --> Final Answer
           |
           no
           v
      next Thought/Action loop
```

- **Insight 1**: user trust increases significantly with step trace + clickable evidence.
- **Insight 2**: parser/input normalization quality directly drives agent reliability.
- **Insight 3**: default one-way behavior avoids common intent mistakes.
- **Insight 4**: crawl/API fallback and transparent error reporting are mandatory for realtime travel use.

### 7. Production Readiness

- **Security**: keys in `.env`, not committed.
- **Guardrails**: `max_steps`, tool constraints, fallback policy.
- **Scalability**: modular split (`agent`, `tools`, `reporting`) ready for cache/queue/schema hardening.
